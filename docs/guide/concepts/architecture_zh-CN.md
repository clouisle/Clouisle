# 系统架构

本文档概述了 Clouisle 的系统架构，解释不同组件如何协同工作，提供企业级 AI Agent 和知识库平台。

## 架构概览

Clouisle 采用现代化、可扩展的架构，前端、后端和基础设施层之间有清晰的分离。

```
┌─────────────────────────────────────────────────────────────────┐
│                         前端 (Next.js 16)                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────────┐ │
│  │  管理后台 │  │  用户平台 │  │   聊天   │  │  认证 (SSO/登录) │ │
│  └──────────┘  └──────────┘  └──────────┘  └──────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                        后端 (FastAPI)                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────────┐ │
│  │  Agent   │  │  工作流  │  │  知识库  │  │   用户与团队     │ │
│  │   引擎   │  │   引擎   │  │         │  │     管理        │ │
```

## 技术栈

### 前端层

**框架**: Next.js 16 with App Router
- **运行时**: Bun 1.0+
- **UI 库**: shadcn/ui (base-vega) + Tailwind CSS
- **语言**: TypeScript
- **状态管理**: React hooks + Context API
- **API 客户端**: Axios with custom interceptors

**主要特性**:
- 服务端渲染 (SSR) 以获得最佳性能
- 多个路由组用于不同的用户体验
- 通过服务器发送事件 (SSE) 实现实时更新
- 国际化 (i18n) 支持

### 后端层

**框架**: FastAPI (Python 3.13)
- **ORM**: Tortoise ORM with AsyncPG
- **任务队列**: Celery + Redis
- **向量数据库**: Qdrant
- **LLM 框架**: LangChain + LangGraph
- **文档处理**: MarkItDown

**主要特性**:
- 全面使用 async/await 实现高并发
- 所有端点的统一响应格式
- 带 i18n 的全面错误处理
- 所有操作的审计日志
- 多渠道通知系统

### 基础设施层

**数据库**: PostgreSQL 16
- 存储所有应用数据（用户、团队、智能体、工作流等）
- ACID 合规性确保数据完整性
- 全文搜索功能

**缓存与队列**: Redis 7
- 会话存储
- Celery 任务代理和结果后端
- 速率限制计数器
- 临时数据缓存

**向量数据库**: Qdrant
- 存储文档嵌入
- RAG 的相似度搜索
- 每个嵌入维度一个集合
- 高效的向量操作

## 组件交互

### 请求流程

#### 1. 用户请求流程

```
浏览器 → Nginx (前端) → Next.js SSR → API 请求 → FastAPI 后端
                                                      ↓
                                                 PostgreSQL
                                                      ↓
                                                   响应
```

#### 2. 聊天请求流程（带 RAG）

```
用户消息 → FastAPI → Agent 引擎 → RAG 检索 → Qdrant
                          ↓                      ↓
                     LLM 适配器                文档
                          ↓                      ↓
                     LLM 提供商 ← 上下文 ← 检索到的块
                          ↓
                     响应 (SSE 流)
                          ↓
                     前端 (实时显示)
```

#### 3. 文档处理流程

```
上传 → FastAPI → Celery 任务 → MarkItDown → 文本提取
                                    ↓
                                  分块
                                    ↓
                              嵌入生成
                                    ↓
                              Qdrant 存储
                                    ↓
                          状态更新 (PostgreSQL)
```

#### 4. 工作流执行流程

```
触发 → FastAPI → Celery 任务 → LangGraph 引擎
                                    ↓
                               节点执行
                               (LLM, 工具, 代码等)
                                    ↓
                               结果存储
                                    ↓
                           SSE 流 (如果实时)
```

## 可扩展性考虑

### 水平扩展

**前端**:
- 无状态 Next.js 实例
- 可扩展到多个副本
- 负载均衡器分配流量

**后端**:
- 无状态 FastAPI 实例
- 可扩展到多个副本
- 会话存储在 Redis（共享）

**Celery Workers**:
- 可扩展到多个 worker
- 基于队列的任务分配
- 不同任务类型的独立队列

**Celery Beat**:
- **必须恰好运行 1 个实例**（定时任务）
- 使用数据库锁防止重复

### 垂直扩展

**PostgreSQL**:
- 增加 CPU/RAM 以提高查询性能
- 连接池提高效率

**Redis**:
- 增加内存以扩大缓存
- 持久化以保证持久性

**Qdrant**:
- 增加内存以支持更大的向量集合
- SSD 以加快磁盘 I/O

## 安全架构

### 身份验证与授权

**多层安全**:
1. **前端**: 路由守卫、基于角色的 UI 渲染
2. **后端**: JWT 令牌验证、权限检查
3. **数据库**: 通过团队隔离实现行级安全

**身份验证方法**:
- 基于密码（使用 bcrypt 哈希）
- SSO（OAuth2、OIDC、SAML、CAS）
- API 密钥（用于程序化访问）

### 数据隔离

**基于团队的多租户**:
- 所有资源属于一个团队
- 用户可以是多个团队的成员
- 查询自动按团队成员资格过滤
- 超级管理员可以访问所有数据

### 审计跟踪

**全面日志记录**:
- 记录所有用户操作
- 更改的前后快照
- IP 地址和用户代理跟踪
- 合规性保留策略

## 性能优化

### 缓存策略

**Redis 缓存**:
- 用户会话（30 分钟 TTL）
- 站点设置（5 分钟 TTL）
- 速率限制计数器（1 小时 TTL）

**数据库索引**:
- 主键（UUID）
- 外键
- 频繁查询的字段（email、username、team_id）

### 异步操作

**后台任务**:
- 文档处理（Celery）
- 邮件发送（Celery）
- 工作流执行（Celery）
- 通知传递（Celery）

**实时更新**:
- 聊天响应的 SSE
- 工作流执行的 SSE
- 未来功能的 WebSocket

## 监控与可观测性

### 日志记录

**应用日志**:
- 结构化 JSON 日志
- 日志级别（DEBUG、INFO、WARNING、ERROR）
- 请求/响应日志
- 错误堆栈跟踪

**访问日志**:
- Nginx 访问日志（前端）
- Gunicorn 访问日志（后端）
- API 端点使用情况

### 指标

**关键监控指标**:
- 请求速率（请求/秒）
- 响应时间（p50、p95、p99）
- 错误率（4xx、5xx）
- 数据库查询时间
- Celery 任务队列长度
- Celery 任务执行时间
- 内存使用
- CPU 使用

### 健康检查

**端点**:
- `/api/v1/health` - 基本健康检查
- `/api/v1/health/db` - 数据库连接
- `/api/v1/health/redis` - Redis 连接
- `/api/v1/health/qdrant` - Qdrant 连接
